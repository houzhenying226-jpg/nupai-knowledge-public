## 正文

## 背景

NuPai CRM IT 测试历经三轮，每轮暴露不同类型的问题。三轮测试的经验总结出一套可复用的方法论。

## 三轮测试特征

| 轮次 | 主要发现 | 根因类型 |
|---|---|---|
| IT1 | 基础功能不可用，API 404/500 | 接口未实现或路由错误 |
| IT2 | 功能能用但数据不对 | 业务逻辑和数据过滤 |
| IT3 | 细节问题：枚举/移动端/AI 功能 | 集成未端对端测试 + 前端映射缺失 |

## 修复工作流

**正确的 Bug 分类方式（优先级从高到低）：**
1. P0：功能完全不可用（API 500 / 页面崩溃）
2. P1：数据错误（显示错误数据 / 过滤失效）
3. P2：显示错误（枚举未翻译 / 布局混乱）
4. P3：体验缺陷（移动端不友好 / 操作步骤多）

**任务分解模板：**
- CC-1（诊断）：读服务器真实代码，不做修改
- CC-2（后端修复）：每个 Fix 修完即 curl 验证
- CC-3（前端修复）：改完 `npm run build` 确认无报错

## 回归清单生成

IT3 修完后应要求生成回归测试 checklist：
```
让 Claude Code 生成 /opt/nupai-crm/tests/regression-checklist.md
覆盖：所有本次修复的功能点的 curl 验证命令
每次部署前跑一遍，输出 ✅/❌ 状态
```

## 经验教训

- IT 测试轮次越多越细，不是越多越坏——每轮都会发现不同层次的问题，这是正常的软件质量迭代。
- 「Claude Code 报告里说没问题」≠ 真的没问题。IT 测试必须有人（振英或真实用户）用浏览器走每个功能路径，截图是最可靠的证据。
- 回归清单把每轮修复的知识固化为可执行验证，是防止同类问题复发的最低成本方式。
