## 背景

## 振英现在的工作流

**启动新版本（每 1-2 周，1-2 小时）：** 与 Claude AI 讨论「v3.X 做什么」→ 走 Skills 口令 A-D → Linear 批量建 Issue 带 ai-fix label。

**日常 7×24（完全脱手）：** 看心跳告警群进度 + 命令机器人 DM 收战报 + D3 审批卡片。

**大版本完成（每 2-4 周，30 分钟）：** 走 Skills 口令 F → 空壳检测 → 口令 G → 蓝图回写。

## 经验教训

- 角色变迁不是自然发生的，需要明确的技术选择（K2 命令机器人 / K9 Sentry 切换 / launchd 自动化）支撑。每一项「不再需要振英手动做」背后都有一个具体的落地项。
- 振英的不满（「还是搬运工」）是最重要的产品需求信号，比任何技术规划文档更直接。把用户痛点转化为可量化的「0 次复制粘贴」目标，是正确的工作方式。
- D3 是「人工必须参与的高风险操作」的最小颗粒，振英的参与保持在最关键的决策点。
