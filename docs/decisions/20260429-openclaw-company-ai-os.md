## 正文

## 背景

v1 部署完结报告将 OpenClaw 定位为「部署助手 + 飞书命令中心」。经 v2 补丁阶段 30+ 小时工作（GPT-5.5 全栈 + before_dispatch 打通 + Workspace 49 文件），振英确认视野升级。

## 定位演进

| 阶段 | OpenClaw 角色 |
|---|---|
| 项目早期（0-3 月） | 部署助手 + 飞书命令中心 |
| 项目中期（3-6 月） | 多 LLM 调度 + 多项目记忆中枢 |
| 项目长期（6 月+） | NuPai AI 军师执行层（CRM/门店/商城多产品线统一记忆 + 跨项目调度） |
| 未来公司规模（1-2 年） | 公司级 AI 操作系统（财务/HR/销售/产品多 agent 协作中心） |

这个定位决定 workspace 设计必须按多项目、多 agent、多团队成员的未来规划，而非单项目工具。

## 振英角色变迁（v1 → 补丁后）

| 维度 | v1 完结 | 补丁完结 |
|---|---|---|
| 飞书 DM /oc 命令 | 命令机器人在跑但 brain 抢路由 | 100% 通，真到 dispatcher |
| LLM 主力 | Codex/Qwen/DeepSeek/Gemini | + GPT-5.5（7 项主力） |
| OpenClaw 角色 | 部署助手 | 公司级 AI 操作系统底座 |

## 经验教训

- 工具的真实定位是在使用中逐渐清晰的，不是一开始就确定的。v1 完结时还是「部署助手」，30 小时后就升级为「公司级底座」——这种视野升级是正常的，设计时预留扩展位（projects/、memory/、skills/）是关键。
- Skills 流程（口令 A-G）与 OpenClaw 分层共存：Skills 负责「做什么」（战略层），OpenClaw 负责「怎么做」（执行层），两者不互相替代。
