# OpenClaw 三模型自动路由白话运营卡与 2026-05-06 部署复盘

> 类型：操作规程 | 项目：openclaw | 风险：L2 | 状态：草稿
> 同步自 Feishu Bitable 表 A | 2026-05-06T04:00:01Z

## 摘要

这张卡把 OpenClaw 三模型自动路由讲成白话运营规则，并沉淀 2026-05-06 这次打通 GPT-5.5 高风险车道的部署经验、踩坑和验收真账。

## 正文

> 记录日期：2026-05-06  风险等级：L2

## 背景
用户希望以后不用手动切 Qwen / DeepSeek / GPT-5.5。
目标不是“谁最强就全上谁”，而是让更好的模型放在最值钱的位置，并且把升级、降级、告警和验收讲成不需要技术背景也能照着看的运营卡。

## 内容
一、白话运营卡
1. 默认不用人判断复杂度，系统自动分。
2. 中文说明、教程、PRD、SOP、总结，优先走 Qwen。
3. 普通开发、读日志、改代码、补测试，优先走 DeepSeek。
4. 高风险任务，直接升到 GPT-5.5。
5. 高风险的定义先写死，不让人临场猜。命中下面任一类，就算高风险：
   - 数据库迁移、回填、删改 schema
   - 权限、认证、支付、安全
   - 上线、回滚、生产事故、Sentry 紧急问题
   - 跨多个模块的大改
   - 同一个问题已经修了两轮还不稳
   - 上线前最终 review 或最终拍板

二、三条固定升级链
1. 普通开发任务：DeepSeek -> GPT-5.5 -> Qwen
2. 中文说明任务：Qwen -> GPT-5.5 -> DeepSeek
3. 高风险任务：GPT-5.5 -> DeepSeek -> Qwen

三、失败和报警规则
1. 单模型失败 1 次：自动重试，不报警。
2. 同一个模型 2 分钟内连续失败 2 次：自动切下一个模型，不报警，只记日志。
3. 同一个任务三层都失败：立刻飞书红色报警，需人工介入。
4. 10 分钟内 5 个任务都发生“主模型降级到备模型”：飞书黄色预警，说明主车道开始不稳。
5. 高风险任务如果 GPT-5.5 连续不可用超过 3 分钟：飞书黄色预警，因为这类任务不该长时间静默降级。

四、给运营/使用人的一句话规则
1. 平时不用手切模型。
2. 写文档、写说明、中文整理，系统多半会给 Qwen。
3. 改代码、补测试、查日志，系统多半会给 DeepSeek。
4. 只要任务涉及迁移、权限、上线、回滚、安全，系统应自动升到 GPT-5.5。
5. 如果发现高风险任务出来的结果像普通实现脑，而不是总工脑，先查日志，不要凭感觉猜。

五、今天这次部署为什么以前没打通
1. 规则层其实早就能识别高风险，但 complex lane 以前是让 NadirClaw 直接去吃 openai-codex。
2. 这条路会碰到 scope 问题，典型报错是缺 `api.responses.write`。
3. 后来证明继续补 ChatGPT Web 模拟链路没有价值，容易碰 Cloudflare、隐藏请求头、接口漂移。
4. 真正可行的办法不是让 NadirClaw 模拟 ChatGPT，而是让 OpenClaw 自己走原生 GPT 入口，NadirClaw 只负责“判路由 + 做协议转换”。

六、今天这次真正打通的关键补丁
1. 开启 OpenClaw 本地 `POST /v1/responses`。
2. NadirClaw 对 `openai-codex/*` 改走 OpenClaw 本机原生桥。
3. 通过 `x-openclaw-model: openai-codex/gpt-5.5` 指定真实 GPT 车道。
4. 将 chat-completions 的 messages/tools/tool_choice 转成 OpenResponses 输入，再把 function_call 结果转回 tool_calls。
5. 对 OpenClaw agent 自己发出的请求，不再把外层已有 tools 继续转发给内层 OpenClaw，避免 tool name conflict。
6. 对流式请求采用“上游缓冲、下游仍给 SSE”，保证 OpenClaw 客户端侧语义稳定。

七、今天踩到的坑
1. 关键词路由如果把 OpenClaw 超长系统提示一起算进去，会把很多任务误判成高风险，所以规则必须优先只看用户消息。
2. 真实端到端请求和裸测请求不是一回事。裸测能通，不代表 OpenClaw agent 链路就一定通。
3. OpenClaw 外层请求本身会带 tool schema；如果桥接层把这套 tools 原样再塞给内层 OpenClaw，会触发 client tool name conflict，然后偷偷 fallback。
4. 200 OK 不等于闭环。必须同时看 OpenClaw 返回、NadirClaw requests.jsonl、stderr 路由日志，确认 selected_model 和 fallback_used。
5. 默认模型如果还停在 `openai-codex/gpt-5.5`，即使 bridge 修好了，也不算“自动路由已启用”；必须把默认模型切回 `nadirclaw/auto`。

八、今天这次的真账验收
1. OpenClaw 本地 `/v1/responses` 直测成功，`x-openclaw-model=openai-codex/gpt-5.5` 能返回 `OK`。
2. NadirClaw 直测成功，`model=openai-codex/gpt-5.5` 文本请求能返回 `OK`。
3. NadirClaw tool-calling 直测成功，能返回 `finish_reason=tool_calls` 和真实 tool call。
4. NadirClaw 流式 SSE 直测成功，tool-calls 和 `[DONE]` 都正常。
5. OpenClaw 真实端到端高风险请求已闭环：默认模型是 `nadirclaw/auto`，请求命中 `high_risk_override` 后，requests.jsonl 记录 `selected_model=openai-codex/gpt-5.5`，且 `fallback_used=null`。

## 结论
这套三模型路由现在可以正式按“自动分工”使用：
普通开发交给 DeepSeek，中文说明交给 Qwen，高风险任务交给 GPT-5.5。
最关键的突破不是“加了更多规则”，而是把 complex lane 改成了 OpenClaw 原生 GPT 入口，终于让“识别高风险”和“真正用 GPT-5.5 执行高风险”这两件事对上了。

## 影响
以后用户不用手动切模型，也不用猜复杂度。
运营上只需要记住两件事：第一，看任务类型；第二，看高风险任务有没有真正命中 GPT-5.5。
技术上最重要的长期经验是：OpenClaw 负责原生 provider 调用，NadirClaw 负责路由，不要再让 NadirClaw 单独模拟 ChatGPT/Codex 后端。

## 下次注意
1. 任何“已打通”都必须做两层测试：裸测和真实 OpenClaw agent 端到端。
2. 任何“高风险已升级”的结论都必须看 `requests.jsonl` 里的 `selected_model` 和 `fallback_used`，不能只看用户回复文本像不像 GPT。
3. 以后如果 GPT-5.5 车道再异常，优先检查四件事：OpenClaw 本地 `/v1/responses` 是否开启、gateway token 是否有效、`x-openclaw-model` 是否被允许、bridge 是否又把外层 tools 转进内层。
4. 白话运营卡要长期跟实现保持一致；模型顺序、告警阈值、fallback 链只要改了，就要同步更新这张卡。
