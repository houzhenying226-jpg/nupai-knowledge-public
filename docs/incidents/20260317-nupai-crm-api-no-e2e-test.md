## 正文

## 背景

IT3 测试中多个后端功能报错（AI 分析失败、公海池领取/分配失败、重置密码失败、提醒中心），Claude Code 在修复报告中标注「API 正常」，但截图明确显示报错。

## 根因分析

Claude Code 「只测了接口存在」（路由注册成功），没有用真实数据测试完整业务流程。「接口存在 ≠ 业务逻辑跑通」。具体问题：
- 数据库字段映射错误，导致查询返回空或报错
- Dify 工作流 URL 配置错误，AI 分析接口 call 失败
- 权限中间件漏加，导致接口返回 403

## 处理方式

修复流程：CC-1 → 连服务器读日志定位失败 API → CC-2 → 逐个接口用真实数据 curl 验证：

```bash
# 必须先登录获取真实 token，再测业务接口
curl -X POST http://39.106.83.79/api/customers/{id}/claim \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json"
```

每个 Fix 修完即测，不批量积累。

## 经验教训

- AI 生成的「修复完成」报告中的「API 正常」结论，必须用真实 token + 真实数据 curl 验证，不能信信口汇报。
- 后端 API 的验收标准：①路由存在 ②真实数据请求返回 200 ③业务逻辑结果正确——三条都要测，只测第一条不算完成。
- CC 任务分解：CC-1（诊断+读代码）→ CC-2（修复+验证）是正确顺序，不能跳过 CC-1 直接修复，因为不读代码不知道真实字段名和错误位置。
