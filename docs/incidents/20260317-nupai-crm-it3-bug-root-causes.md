## 正文

## 背景

NuPai CRM IT 第三轮测试发现 20 个 Bug，通过截图人工分类后归纳为四类根因。此分类为后续分批修复提供了方向。

## 四类根因

**第一类：后端 API 集成未端对端测试（占比最大）**

AI 分析失败、公海池领取/分配失败、重置密码失败、提醒中心按钮失败——API 代码写了，但和数据库/Dify 的连接没有端对端测试。只测了「接口存在」，没测「接口跑通」。代表 Bug：B2/B5/B6/B8/B13/B14。

**第二类：枚举值前后端约定不一致**

规模显示「large」、拜访类型显示「onsite」、情绪显示「neutral」——前端忘记做中文映射，后端存的是英文枚举（正确），前端直接渲染原始值。竞品档案显示 `删除` 是中文字符被多转义一次，典型的 AI 生成后未验证代码。代表 Bug：B1/B3/B20。

**第三类：数据过滤条件写错**

商机拜访显示所有拜访、KPI 全部 0、月度计划统计错误——功能能运行但数据不对。通常是 WHERE 条件漏写、字段名用错（user_id vs sales_id）、时区问题（UTC 存储但查询用 CST）。代表 Bug：B7/B10/B11。

**第四类：移动端从未真机测试**

选项卡无横向滚动、双列布局在小屏无 fallback——设计以 PC 为主，移动端是附带适配，从未经过真机测试。代表 Bug：B18/B19。

## 经验教训

- 这 20 个 Bug 说明系统经历了「能跑起来」的验收，但没经历「每个功能完整走一遍」的集成测试。不是架构问题，是测试覆盖率问题。
- AI 生成代码必须验证中文字符处理，特别是 JSON 序列化/反序列化路径容易出现双转义。
- 每轮 IT 测试修完后，应让 Claude Code 生成回归测试 checklist，部署前自动运行，避免重复出现同类问题。
